# PlateBlur 当前车牌识别原理与算法说明

## 1. 目标

PlateBlur 不是做“车牌内容识别”优先的系统，而是做“车牌区域检测 + 一键脱敏”优先的系统。

产品目标是：

- 用户用 iPhone 拍照或导入车辆图片
- App 在本地快速找到车牌区域
- 用户几乎不用手动修图
- 直接输出已经打码的结果图

所以当前算法链路的核心任务是：

1. 尽可能高召回地找到车牌框
2. 在复杂街景、小目标、远距离场景下尽量减少漏检
3. 如果主模型漏掉，再用 OCR 和几何候选兜底

## 2. 总体流程

当前 App 的识别流程是一个多阶段检测管线：

1. 先跑基础 Core ML 车牌检测器 `LicensePlateDetector`
2. 如果“增强识别”开启，再额外跑小目标强化检测器 `LicensePlateDetectorSwiss`
3. 将两路检测结果做去重合并
4. 再跑 OCR 文本识别，用国家牌照规则匹配疑似车牌文本
5. 如果前几步都没有可靠结果，再跑矩形候选兜底
6. 最终把检测框交给打码渲染器输出遮挡图

这条链路的关键设计思想是：

- 主模型负责大部分正常场景
- 增强模型负责远距离、小目标、复杂街景
- OCR 负责“模型没看见，但文字比较明显”的情况
- 矩形兜底负责最后的恢复路径，而不是主检测方案

## 3. 基础检测器

基础检测器的入口是 `CoreMLPlateDetector`，默认加载 `LicensePlateDetector.mlmodelc`。

它做的是目标检测，不是字符识别。模型输出的是：

- 车牌框位置
- 每个框的置信度

当前实现里，基础检测器使用：

- 输入尺寸：`640 x 640`
- 全图阈值：`0.10`
- 滑窗阈值：`0.10`
- NMS IoU：`0.45`

基础检测器适合这些情况：

- 近中距离车辆图
- 单车或少量车辆
- 车牌在图里不算特别小
- 普通门店拍照或二手车展示图

## 4. 增强识别

“增强识别”是当前这版为困难场景加上的第二路模型。

虽然模型文件名仍然叫 `LicensePlateDetectorSwiss`，但在产品含义上，它现在不是“瑞士专用模型”，而是“更偏小目标和困难街景的增强检测器”。

它之所以最初从瑞士公开集方向微调出来，是因为那类数据里有大量：

- 远距离车牌
- 画面中车牌面积很小
- 多车混行
- 城市道路抓拍
- 复杂背景

这些恰好和真实手机拍照里最容易漏检的情况高度重合。

当前增强检测器使用：

- 输入尺寸：`640 x 640`
- 全图阈值：`0.20`
- 滑窗阈值：`0.20`

增强识别开启时，App 会同时跑：

- 基础检测器
- 增强检测器

然后把两边的框合并，再做去重。

这意味着：

- 正常图片仍然有基础模型兜住
- 难图会多一次更强的小目标检测
- 用户不需要知道“这是不是瑞士图”

## 5. 为什么要做滑窗推理

困难场景里的核心问题，不是国家差异，而是目标尺度。

如果原图里车牌很小，直接整张图缩放到 `640 x 640` 以后，车牌可能只剩十几像素宽，检测器很容易直接看不见。

所以当前实现不是只做“整图一次推理”，而是做：

- 全图推理
- 局部滑窗推理
- 两者合并

当前滑窗参数：

- `tile_size = 640`
- `tile_overlap = 0.40`

这样做的好处是：

- 全图推理保留上下文
- 滑窗推理放大小车牌局部区域
- 对远距离小目标召回提升明显

这也是这次困难街景准确率提升最重要的算法改动之一。

## 6. 合并与去重

两路模型和滑窗推理都可能给出重复框，所以当前系统在输出前会做去重。

核心思路是：

- 按置信度排序
- 对重叠过高的框只保留一个
- 过滤掉无意义的小框和越界框

这样可以避免：

- 同一块车牌被画出多个框
- 滑窗边界产生重复结果
- 导出时出现多个重叠打码层

## 7. OCR 兜底

OCR 不是主算法，而是兜底恢复路径。

当主模型没给出足够结果时，系统会再做一次 Vision 文本识别，然后根据牌照规则匹配疑似车牌文本。

当前 OCR 规则支持：

- 德国
- 荷兰
- 瑞士

这里要强调：

- OCR 地区规则只影响文本匹配兜底
- 不影响增强识别模型是否运行

所以设置里的“车牌地区”本质上是：

- OCR 规则选择器
- 不是主模型切换器

## 8. 矩形候选兜底

如果主模型和 OCR 都没找到结果，系统最后还会尝试矩形候选检测。

它不是学习型模型，而是几何恢复路径，用来找“看起来像牌照区域的矩形块”。

这个阶段的定位很明确：

- 用于防止完全空结果
- 方便用户手动微调
- 不应该替代训练好的车牌检测器

## 9. 坐标与渲染

识别完成后，系统不会直接输出文本，而是把检测框交给打码渲染器。

渲染阶段支持：

- 纯色遮挡
- 马赛克
- 模糊

并且会在真正绘制前做这些事情：

- 坐标归一化
- 适配原图像素尺寸
- 修正图像方向
- 在导出渲染时翻转到正确图像坐标系
- 根据用户设置做外扩 padding

这一步很重要，因为检测框和图像绘制坐标系不一致时，就会出现“看起来识别到了，但打码层画错位置”的问题。

## 10. 为什么不用 OCR 直接识别车牌

因为当前产品的目标不是读出车牌号，而是高效打码。

单独依赖 OCR 会有几个明显问题：

- 远距离时字符根本看不清
- 反光、遮挡、压缩图下字符不稳定
- OCR 更适合文本读取，不适合先找小目标位置

所以对 PlateBlur 来说，更合理的顺序是：

1. 先检测车牌区域
2. 再做遮挡
3. OCR 只作为辅助兜底

## 11. 当前优势

这套算法的当前优势是：

- 完全本地运行，适合 iPhone 即拍即处理
- 双检测器 + 滑窗，对困难街景比单模型稳
- OCR 和矩形兜底让“完全空结果”的概率更低
- 用户还可以手动补框，保证最后一定能导出

## 12. 当前局限

当前版本仍然有这些局限：

- 增强模型虽然能显著改善难图，但不是全能模型
- 如果把增强模型当成唯一主模型，会拖累普通基准表现
- OCR 在模糊、远距离、小牌照情况下帮助有限
- 矩形兜底只能作为恢复路径，准确率不如训练好的模型

所以现阶段最合理的产品策略不是“单模型统一解决一切”，而是：

- 基础检测器负责正常图
- 增强识别负责困难图
- OCR 与矩形负责兜底
- 手动补框负责最终交付闭环

## 13. 当前最贴近产品需求的理解

如果用一句话概括当前算法：

PlateBlur 现在采用的是“本地双模型检测 + 滑窗增强 + OCR/几何兜底 + 手动修正”的车牌脱敏链路，目标是让用户在手机上拍完车图后，尽量一键完成打码，而不是去做车牌内容识别或车牌查询系统。
